# 디시인사이드 특정 갤러리 데이터 아카이빙 (ETL 파이프라인) 아키텍처 명세서

## 1. 프로젝트 개요
- 목적: 특정 디시인사이드 갤러리(예: maplerpg)의 게시글 데이터(본문 포함)를 주기적으로 스크래핑하여 Google Sheets에 적재(아카이빙).
- 구성: 
  1) 실시간 수집 봇 (GitHub Actions 기반 - scraper.py)
  2) 과거 데이터 온보딩 스크래퍼 (로컬 PC 기반 - onboarding_scraper.py)
- 보안 규칙: API 키 및 인증 정보는 소스코드에 하드코딩하지 않고 환경 변수(GCP_CREDENTIALS)를 사용. `.env` 파일 및 인증 파일은 `.gitignore`에 등록하여 GitHub 퍼블릭 리포지토리 노출을 원천 차단함.

## 2. 모듈 1: 실시간 수집 봇 (GitHub Actions - scraper.py)
- 구동 환경: GitHub Actions (Ubuntu 환경)
- 스케줄링: Cron 표현식 (`0 * * * *`)을 사용해 매 1시간 정각마다 실행.
- 핵심 작동 논리 (증분 수집):
  - 구글 시트의 A열(글 번호) 데이터를 Set 자료구조로 캐싱.
  - 최신 1페이지부터 스크래핑을 시작.
  - 리스트를 파싱하다가 '캐싱된 Set에 이미 존재하는 글 번호'를 만나면, 직전 타임에 수집을 완료한 지점으로 판단하여 탐색(크롤링)을 즉시 종료(Break).
  - 첫 실행 감지: 시트에 수집된 글 번호가 0개일 경우, 무한루프 및 밴 방지를 위해 오직 1페이지만 수집하고 안전하게 종료.
  - 수집된 '새 글' 목록은 시간순(오래된 순 -> 최신 순)으로 정렬(reverse)하여 구글 시트 맨 아래에 일괄 Append.

## 3. 모듈 2: 과거 데이터 온보딩 스크래퍼 (로컬 PC - onboarding_scraper.py)
- 구동 환경: 사용자의 로컬 환경 (수동 실행)
- 로컬 실행 이유: 수십~수백 페이지의 과거 글을 단기간에 수집할 경우 디시인사이드 서버의 DDoS 방어 시스템에 의해 GitHub 클라우드 IP가 영구 밴될 위험이 있음. 따라서 로컬 IP에서 장시간의 딜레이를 가지고 안전하게 수집하기 위함.
- 핵심 작동 논리:
  - 사용자로부터 시작 페이지와 종료 페이지를 Input으로 받음.
  - 기존 구글 시트의 A열과 대조하여 이미 존재하는 글은 스킵(Continue) 처리하되, 실시간 봇처럼 전체 루프를 중단시키지는 않음 (중간 이빨 빠진 데이터 수집 가능).

## 4. 안티 봇(차단 우회) 스텔스 전략 공통 적용 사항
- Headers 위장: User-Agent, Accept 등을 일반 브라우저와 완벽히 동일하게 하드코딩.
- SSL 경고 우회: 디시 서버 특성상 인증서 오류로 접속이 튕기는 것을 막기 위해 `urllib3.disable_warnings` 및 `verify=False` 옵션 적용.
- 인간적인 랜덤 딜레이:
  - 본문 조회 시 `time.sleep(random.uniform(1.5, 3.0))` 적용.
  - 페이지 전환 시 1~2초 또는 그 이상의 랜덤 딜레이 강제 삽입.
- 타임아웃 방어: `timeout=6` (로컬은 10) 옵션을 주어 멈춤 현상 방지 및 상태 코드가 200이 아닐 경우 예외 처리 후 안전하게 프로세스 종료.

## 5. 데이터 스키마 (Google Sheets 구조)
1행을 헤더로 사용하며, A열부터 I열까지 아래의 순서로 적재됨.
[A: 글번호, B: 제목, C: 본문, D: 작성자, E: 날짜(YYYY-MM-DD HH:MM 형식 스마트 변환), F: 링크, G: 댓글수, H: 조회수, I: 추천수]
